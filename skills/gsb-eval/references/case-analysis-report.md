# Case 分析报告写作规范

本规范用于 GSB 评估后的 good case / bad case 专项分析。适用场景包括：

- 用户要求「case 分析」「bad case 分析」「good case 分析」
- 用户希望汇总模型当前问题、问题分布和代表 case
- 用户希望从评估结果中提炼可驱动模型、prompt、商品召回、商品卡或评估口径优化的结论

case 分析报告不是 case dump，也不是只统计 good / bad 数量。报告的目标是让负责人能判断：

- 当前稳定问题或稳定能力是什么
- 哪些是偶发噪声，哪些能 drive 优化
- 不同模型版本各自好在哪里、差在哪里
- 用户/评审为什么这么判断
- 模型回复中具体哪里好、哪里坏
- 后续应该如何改，以及哪些 case 应进入 regression

核心原则：

```text
从 case 中提炼问题簇或能力簇；
用 comment 证明问题；
用双版本回复对比让问题具象化；
最后给出可执行改进动作。
```

## 0. 行文风格偏好

case 分析报告要优先降低阅读负担。读者通常不是来读完整论文，而是要快速判断「问题是什么、证据是什么、后续怎么改」。

写作要求：

- **更简洁**：能一句话说明的，不写三句话。
- **多分段**：一段只讲一个判断，避免长段落堆叠。
- **少用绕口句式**：避免高频使用「不是……而是……」「并非……而是……」这类转折句，直接说重点。
- **少做过度总结**：case 已经能证明的问题，不要再用大段话重复抽象归纳。
- **合并同类问题**：如果决策题、对比题都指向「0403 过早推荐商品、选购思路不如 baseline 清晰」，可以合并成一个问题簇，不要硬拆成两个重复章节。
- **不要让 baseline 优点散落各处**：baseline 的共性优势应集中写在问题簇开头或总表里；case 后只补充该 case 特有的差异。
- **表格优先承载横向信息**：版本优缺点、问题簇、改进动作适合用表格；正文只解释最关键的判断。
- **正文加粗克制**：只加粗核心结论和少数关键判断，不给标签、普通 bullet、整列表格大面积加粗。

## 1. Case 筛选原则

不要把所有 bad / good case 都放进正文。正文优先放高信号 case。

### 应进入正文

| 类型 | 说明 |
| --- | --- |
| 多评审一致或票差明显 | 问题信号稳定 |
| comment 指向具体问题 | 能解释为什么好/坏 |
| 两版差异能说明能力短板 | 可以 drive 模型或策略优化 |
| 两版都不好 | 即使暂时不知道怎么改，也先保留给人工判断 |
| 表达风格 / 结构问题 | 如果影响决策效率，也先保留 |
| 安全、健康、老人儿童、专业设备等高风险场景 | 样本少也值得关注 |
| 可沉淀为 regression 的样本 | 后续版本必须回归验证 |

### 可放观察池或排除

| 类型 | 说明 |
| --- | --- |
| 弱票差且无具体 comment | 信号不足 |
| 单个用户偏好 | 暂时无法归因 |
| 无法定位具体问题 | 不适合正文展开 |
| 重复同类问题 | 可放观察池，不必全部展开 |

### below 口径

如果报告使用 `below`：

```text
below 指某个评审核员/标注人员认为该题整体质量低于预期。
它不是指某个模型，也不是指任意模型打分。
```

## 2. 总体结论写法

不要把“多少个 case”当结论。数量只辅助说明分布。

好的总体结论应该包含：

- 当前版本的稳定短板
- 对照版本的可借鉴点
- 对照版本自身的问题
- 两版共同问题
- 明确的优化目标

示例写法：

```markdown
0403RL 在导购场景里容易过早推荐商品或直接下结论，用户真实纠结没有被充分承接。

baseline 更常先拆问题、给判断框架、再推荐商品，因此更有决策帮助。

baseline 的稳定问题是冗长、重复、商品卡过多，偶尔推荐过专业或过度劝退。

优化目标：保留 0403RL 的简洁、直接和商品承接能力，补上问题拆解、对比框架、风险边界、选购依据。
```

避免写成：

```markdown
本次共有 37 个 bad case，其中 13 个是商品推荐问题，9 个是需求理解问题。
```

这种数量统计可以放分布表，但不能当核心结论。

### 行文风格

case 分析报告要优先保证阅读效率。默认风格：

- 句子短，结论直接。
- 多分段，降低连续阅读压力。
- 少用“不是……而是……”这类绕行句式，直接写判断。
- 少做泛泛归纳；归纳必须能压缩多个 case 的共同问题。
- baseline / 对照版本的优点集中写，不要散落在每个 case 后反复出现。
- 能合并的问题簇要合并。例如“决策题承接不足”和“对比题框架不清”都可能指向“选购思路不清晰”，先合并成更高层问题，再在下层拆具体表现。
- 避免每个 case 都重复“具体问题 / 为什么 baseline 更好 / 驱动改进”。更推荐“问题簇结论 + 证据表 + 必要原文摘录”的结构。
- 原文摘录只保留能直接看出问题的片段。摘录后只写一两句判断，避免长篇解释。

推荐表达：

```markdown
0403RL 的主要问题集中在三类：

- 选购思路过急：决策题、对比题里经常很快进入商品推荐或单边结论。
- 对比框架不清晰：用户给出两个候选项时，回答更像分别介绍，不像帮助用户选。
- 商品和风险边界不稳：推荐少、商品卡偏离约束、版本过时、风险提示不足。
```

不推荐表达：

```markdown
0403RL 的主要问题不是不会回答，而是在导购任务中常常缺少对用户真实纠结的承接，因此 baseline 的优势不是单纯更长，而是能更系统地拆解问题……
```

## 3. 推荐报告结构

```markdown
# Case 分析报告

## 1. 筛选口径
- bad / good case 来源
- below / exceeds / GSB bad 等标签解释
- 哪些进入正文
- 哪些放观察池
- 哪些排除

## 2. 总体结论
- 当前版本稳定问题
- 对照版本可借鉴点
- 对照版本自身问题
- 两版共同问题
- 可驱动改进方向

## 3. 问题簇一：问题名称
### 共性问题
- 该类 query 的本质
- 当前版本常见问题
- 对照版本可借鉴点
- 对照版本也有的问题
- 用户/评审 comment 共性

### Case 1
- Case ID
- 评估结果
- 用户 comment
- 当前版本片段
- 具体问题
- 对照版本片段

> 注意：如果「为什么 baseline 更好」「驱动改进」已在问题簇开头集中说明，case 后不要重复写长段总结。

## 4. 问题簇二
...

## 5. 两版共同不好的 case
- 共同问题
- 为什么要保留
- 可能驱动的系统策略

## 6. Regression 建议
- case_id
- 问题类型
- 为什么进回归集

## 7. 观察池
- case_id
- 问题简述
- 暂不展开原因

## 8. 人工标注建议
- 哪些地方需要人工高亮
- 哪些判断需要业务确认
```

## 4. 问题簇写法

问题簇名称要直接、具体、能指导改进。

推荐：

```text
决策题没有真正接住用户纠结
对比题缺少稳定的“差异框架 + 最终建议”
商品推荐和商品卡质量不稳定
专业、健康、安全类问题的信息深度和风险边界不足
两版共同不好，暴露系统策略问题
```

不推荐：

```text
用户意图与场景
信息完整性
结构与可读性
其他问题
```

如果必须用抽象分类，下面必须解释它具体指什么。

## 5. 问题簇共性分析写法

每个问题簇先写共性，不要直接进入 case。
共性部分要承担「归并」职责：把多个 case 的共同问题讲清楚，避免后面每个 case 都重复说明 baseline 优点。

推荐结构：

```markdown
这类 query 的核心是：用户已经看上了某个东西，但在为缺点、价格、闲置风险或场景适配纠结。模型要帮助用户做决策。

0403RL 的常见问题：

- 过早进入商品推荐。
- 选购思路不如 baseline 清晰。
- 对比方式不够稳定，表格化表达少。

baseline 的可借鉴点：

用户 comment 的共性：

- “没有重点回答是否需要购买”
- “总结了用户最深的困惑”
- “从实际出发更贴合”
- “承认喜欢的心理并给理性支撑”

baseline 也有的问题：

- 有时过长、劝退感强。
- 偶尔引入用户没提过的细节。
```

## 6. Case 展示规范

case 展示必须做双版本对比。只放当前版本原文不够。
case 展示要尽量短。不要把每条 case 都写成完整小报告；case 的主要作用是让读者看到问题如何发生。

标准结构：

````markdown
### Case：国企终面要不要买 800-900 元拉夫劳伦 Polo

Case ID：`xxx`

评估结果：baseline 胜，票型 `baseline 7 / 0403RL 1 / similar 1`，聚合分 `-10`。

用户 comment：

- 0403RL 缺点：没有重点回答用户问题，是否需要购买
- baseline 优点：开头给出明确结论，并用多维度对比展开，用户一眼能看懂判断依据

0403RL 片段：

```markdown
...
```

具体问题：

- 用户问的是“有必要花八九百买吗”，0403RL 没有在开头明确回答。
- 商品卡把拉夫劳伦包装成“面试适配”，削弱了国企终面应正式稳妥的关键判断。
- 后续建议变成 Polo 衫购买建议，而不是终面着装决策。

baseline 片段：

```markdown
...
```

对比结论：

- 先回答“有没有必要买”，不是先推荐商品。
- 把场景风险讲清楚：国企终面、正式度、单次面试、预算机会成本。

改进动作：

- “值不值得买”类 query 首段必须给出明确结论。
- 结论格式建议：`不建议/建议/视情况` + `最关键原因` + `替代方案`。
- 若用户问的是决策，商品卡不能早于决策框架出现。
````

## 7. 片段摘录长度

片段不要太短，也不要全文堆进去。

推荐：

- 保留开头结论
- 保留关键推理
- 保留问题所在的句子或段落
- 保留必要的商品卡位置，用 `[商品卡]` 代替原始标签
- 中间无关展开用 `...` 省略
- 当前版本和对照版本都要放片段

不推荐：

- 只摘一句，看不出问题
- 放完整长回答，淹没分析
- 只放当前版本，不放对照版本

## 8. 评估结果格式

统一格式：

```markdown
评估结果：baseline 胜，票型 `baseline 5 / 0403RL 0 / similar 0`，聚合分 `-7`，below_count `1`。
```

要求：

- 顺序固定：`baseline / 0403RL / similar`
- 0 票也写出来
- 聚合分保留
- 有 below 时写 `below_count`
- 不要出现有的地方写 `票型 baseline 5`、有的地方写 dict 的混乱格式

## 9. Comment 使用规范

comment 要提炼共性，也要放在具体 case 中。
优先合并同义 comment，不要把所有 comment 原样平铺。case 内只放能解释胜负或问题的 comment。

### 问题簇层面

用于总结稳定问题：

```markdown
用户 comment 的共性：
- “没有重点回答是否需要购买”
- “推荐商品种类较少”
- “两边都选错了”
```

### Case 层面

用于解释为什么这个 case 被保留：

```markdown
用户 comment：
- 0403RL 缺点：推荐商品种类较少，且推荐类型很奇怪
- baseline 优点：场景分析更明确，推荐商品种类多
- baseline 缺点：很多模块重复推荐
```

如果多个 comment 表达同一意思，可以合并：

```markdown
baseline 优点：开头给出明确结论，并用多维度对比展开，用户一眼能看懂判断依据
```

## 10. 具体问题定位

每条 case 必须尽量指出具体问题。
具体问题定位要短，通常 1-3 条即可。不要在每个 case 后重复问题簇层面的完整结论。

好的写法：

```markdown
具体问题：
- 用户问的是“有必要花八九百买吗”，0403RL 没有在开头明确回答“不要买/不建议为单次面试买”。
- 商品卡把拉夫劳伦包装成“面试适配”，削弱了“国企终面应更正式稳妥”的关键判断。
- 后续建议变成 Polo 衫购买建议，而不是终面着装决策。
```

差的写法：

```markdown
具体问题：
- 结论不清晰。
- 不够贴合用户。
```

如果无法可靠判断，就写：

```markdown
此处需要人工二次标注。当前只保留原始片段和 comment。
```

## 11. 两版共同问题

两版都不好时，不要因为“当前不知道怎么改”就排除。先进入正文或观察池。

写法：

```markdown
### Case：手机充电口小电扇

共同问题：两版都可能没有严格匹配“充电口可用”的硬约束。

改进动作：
- 商品召回前先抽取硬约束：`小米17`、`充电口`、`小电扇`、`150mA/300mA`。
- 如果召回商品无法确认接口兼容性，应先说明不确定，不能硬推。
```

## 12. 改进动作写法

改进动作必须可执行，不能写空话。

差：

```markdown
提升理解能力。
优化推荐质量。
加强专业性。
```

好：

```markdown
值不值得买类 query 首段必须输出：建议 / 不建议 / 视情况 + 关键原因 + 替代方案。
```

```markdown
对比题必须包含：候选项识别、用户约束、核心差异表、明确推荐、不确定信息说明。
```

```markdown
电子产品推荐必须校验：当前主流型号、上市时间、用户经验水平、预算、是否性能过剩。
```

```markdown
健康/老人/儿童类 query 必须先给风险边界和就医/专业建议，再推荐商品。
```

## 13. Regression 建议

报告最后要沉淀 regression case。

格式：

```markdown
| 问题 | Case ID | 原因 |
| --- | --- | --- |
| 决策题不回答“要不要买” | `ca8a3c08` | 票差大，问题明确 |
| 情绪价值 vs 功能硬伤 | `0123570a` | 能测共情和决策边界 |
| 对比题无最终建议 | `42fc243b` | 宠物安全 + 商品对比 |
```

进入 regression 的优先级：

- 票差大
- comment 明确
- 能代表稳定问题
- 两版都不好但有系统修复价值
- 高风险场景
- 商品召回/商品卡问题可复现

## 14. Good Case 复用同一思路

good case 不要只展示“模型回答好”。应写：

- 好在哪里
- 用户/评审如何 comment
- 相比另一版好在哪里
- 这种能力是否稳定
- 是否值得迁移到 prompt / policy / 训练数据
- 是否要进入 positive regression

结构可以复用：

```markdown
能力簇
用户 comment 共性
当前版本好在哪里
对照版本差在哪里
case 对比
可复用能力
后续如何保持
```

## 15. 最终验收标准

一份高质量 case 分析报告应该满足：

- 读者不用打开原始数据，也能理解问题
- 每个问题簇都能对应到明确 case
- 每个 case 都有用户 comment 和双版本片段
- 每个 case 都指出具体哪里好/坏
- 报告同时分析当前版本、baseline、两版共同问题
- 报告能导出 action 和 regression
- 不被低信号噪声淹没，也不漏掉两版共同失败和表达结构类问题
